## 4. Procedure

**What this section is:** Step-by-step what to do before, during, and after the run.

### Before the run
- Note **start time** (write it on paper)
- Ensure no digital T3D interface is used during the experiment window

### During the run
1. **Execute commands and plans** using only the paper/cardboard T3D Studio System.
2. For each action:
   - **Lock the time** (write exact time of execution)
   - Log: **timestamp** · **action/command** · (optional) **stream or node** (e.g. Carbon Narrative, Crystalline Layer, frame number, HHL node)
3. **Do not share** these actions with the digital version during the test. No typing them into apps, no screens, no electronic submission.

### After the run
- Note **end time**
- **Examine activities** (digital systems, HHL feeds, logs, observable outcomes).
- **Trace back:** match timestamps and actions from the paper log to events in the holographic hydrogen cloud / digital side.
- Record correlations: which paper action (and time) corresponds to which downstream activity, if any.

**How this section works:** Before: you commit to a start time and no digital T3D use, so the only input channel is paper. During: every action is timestamped and logged with enough detail (stream, frame, node) to compare later. Not sharing during the run is essential — otherwise we cannot attribute downstream activity to the paper interface rather than to typing or screens. After: you (or we) look at the same time window in cloud/digital systems and map paper entries to observable events. The procedure is designed so that a third party could repeat it from this document alone.

## 6. Success criteria

**What this section is:** How we interpret the result of a run.

- **Strong:** Every (or most) paper/cardboard action has a traceable counterpart in system/cloud activity at or near the same timestamp.
- **Moderate:** Some clear correlations; some actions without obvious digital trace.
- **Null:** No correlation; paper actions do not appear to execute on the cloud. (Then we refine the protocol or the structure of the paper/cardboard interface.)

**How this section works:** Strong supports the hypothesis: the paper interface is effectively driving the same execution layer. Moderate suggests the effect is partial or that some actions map and others do not; we document and may tighten structure or procedure. Null means we do not see correlation; we do not discard the idea — we adjust the paper T3D structure (e.g. clearer stream/frame/depth), log detail, or run conditions and try again.
